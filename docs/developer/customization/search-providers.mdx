## Overview

Spree uses a pluggable search provider architecture. The search provider handles product search, filtering, sorting, and faceted navigation. By default, Spree uses SQL-based search (ILIKE queries via Ransack). You can replace it with an external search engine like [Meilisearch](/integrations/search/meilisearch) for typo tolerance, relevance ranking, and faster faceted search.

## Architecture

```
Store API Request
  │
  ├─ AR Scope (security + visibility)
  │   store.products.active(currency).accessible_by(ability)
  │
  └─ Search Provider (search + filter + facets)
       ├─ Database (default): ILIKE + Ransack + FiltersAggregator
       └─ Meilisearch: one API call for search + filter + facets
```

The controller builds a base ActiveRecord scope for **security and visibility** (store scoping, active products, authorization), then delegates **everything else** to the search provider — text search, structured filtering, sorting, pagination, and facet aggregation.

## Configuration

Set the search provider in `config/initializers/spree.rb`:

```ruby
# Default — SQL-based search (no external dependencies)
Spree.search_provider = 'Spree::SearchProvider::Database'

# Meilisearch — recommended for production
Spree.search_provider = 'Spree::SearchProvider::Meilisearch'
```

## Built-in Providers

### Database (default)

`Spree::SearchProvider::Database` wraps the existing Ransack + ILIKE search. No external dependencies. Suitable for development and small catalogs (< 10,000 products).

- Text search: ILIKE on product name + variant SKU
- Filtering: Ransack query params (price, options, categories, stock)
- Facets: `FiltersAggregator` with per-option-value counts
- Sorting: Ransack + custom scopes (price, best_selling)

### Meilisearch

`Spree::SearchProvider::Meilisearch` uses [Meilisearch](https://www.meilisearch.com/) for search, filtering, and faceting in a single API call. Recommended for production. See the [Meilisearch integration guide](/integrations/search/meilisearch) for setup instructions.

## Building a Custom Search Provider

Create a class that inherits from `Spree::SearchProvider::Base` and implements `search_and_filter`:

```ruby
# app/models/my_app/search_provider/typesense.rb
module MyApp
  module SearchProvider
    class Typesense < Spree::SearchProvider::Base
      def search_and_filter(scope:, query: nil, filters: {}, sort: nil, page: 1, limit: 25)
        # 1. Query your search engine
        results = typesense_client.collections['products'].documents.search({
          q: query || '*',
          query_by: 'name,description,sku',
          filter_by: build_filters(filters),
          sort_by: build_sort(sort),
          page: page,
          per_page: limit,
          facet_by: 'category_ids,in_stock,price'
        })

        # 2. Get product IDs from search results
        ids = results['hits'].map { |h| h['document']['id'] }

        # 3. Intersect with AR scope for security/visibility
        products = ids.any? ? scope.where(id: ids).in_order_of(:id, ids) : scope.none

        # 4. Return a SearchResult
        Spree::SearchProvider::SearchResult.new(
          products: products,
          filters: build_facet_response(results['facet_counts']),
          sort_options: %w[price -price name -name -available_on],
          total_count: results['found']
        )
      end

      # Index a product (called via background job on product save)
      def index(product)
        document = Spree::SearchProvider::ProductPresenter.new(product, store).call
        typesense_client.collections['products'].documents.upsert(document)
      end

      def remove(product)
        remove_by_id(product.id)
      end

      def remove_by_id(product_id)
        typesense_client.collections['products'].documents[product_id].delete
      end

      def reindex(scope = nil)
        scope ||= store.products
        scope.find_in_batches(batch_size: 500) do |batch|
          documents = batch.map { |p| Spree::SearchProvider::ProductPresenter.new(p, store).call }
          typesense_client.collections['products'].documents.import(documents, action: 'upsert')
        end
      end
    end
  end
end
```

Then register it:

```ruby
# config/initializers/spree.rb
Spree.search_provider = 'MyApp::SearchProvider::Typesense'
```

### Provider Contract

| Method | Required | Description |
|--------|----------|-------------|
| `search_and_filter(scope:, query:, filters:, sort:, page:, limit:)` | Yes | Search, filter, sort, paginate, and return facets. Must return a `SearchResult`. |
| `index(product)` | No | Index a single product. Called via background job on create/update. No-op for database provider. |
| `remove(product)` | No | Remove a product from the index. Called via background job on destroy. |
| `remove_by_id(product_id)` | No | Remove by ID (product may already be deleted). |
| `reindex(scope)` | No | Bulk reindex. Called via `rake spree:search:reindex`. |

### SearchResult

`search_and_filter` must return a `Spree::SearchProvider::SearchResult`:

```ruby
Spree::SearchProvider::SearchResult.new(
  products: ar_relation,      # ActiveRecord::Relation — paginated products
  filters: [...],             # Array of filter hashes (facets)
  sort_options: [...],        # Array of sort option strings
  total_count: 150            # Integer — total matching products (before pagination)
)
```

### ProductPresenter

Use `Spree::SearchProvider::ProductPresenter` to serialize products for indexing. It handles multi-currency prices, translations, option values, categories (with prefixed IDs), and stock status:

```ruby
presenter = Spree::SearchProvider::ProductPresenter.new(product, store)
document = presenter.call
# => {
#   id: 42, prefixed_id: "prod_abc", name: "Blue Shirt", slug: "blue-shirt",
#   status: "active", sku: "SHIRT-001", in_stock: true,
#   category_ids: ["ctg_abc", "ctg_def"], category_names: ["Clothing", "Shirts"],
#   option_values: ["Small", "Medium", "Large", "Blue", "Red"],
#   price_USD: 29.99, price_EUR: 27.50, compare_at_price_USD: 39.99,
#   name_en: "Blue Shirt", name_fr: "Chemise Bleue", ...
# }
```

### Indexing Lifecycle

Products include the `Spree::SearchIndexable` concern which provides:

- **Automatic indexing** — background jobs enqueue on `after_commit` (create/update/destroy). Jobs are skipped for the Database provider.
- **Manual indexing** — `product.add_to_search_index` (synchronous, inline)
- **Manual removal** — `product.remove_from_search_index` (synchronous, inline)
- **Preview** — `product.search_presentation` returns the hash that would be indexed
- **Bulk reindex** — `rake spree:search:reindex`

### Important: Prefixed IDs

Always use prefixed IDs (e.g., `ctg_abc`, `prod_xyz`) when indexing association references. Never use raw database IDs — Spree supports UUID primary keys.
